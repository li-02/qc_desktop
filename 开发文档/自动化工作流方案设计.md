## 1. 概述

### 1.1 目标

自动化工作流模块的目标是让用户能够将应用现有的数据处理功能（异常检测、缺失值插补、通量分割、相关性分析、数据导出）编排成**可保存、可复用、可自动执行**的线性流水线。

> **v2.0 重要变更**：工作流重构为**独立的流程模板**，与数据集解耦。工作流创建时不再绑定数据集，而是在执行时选择目标数据集和初始版本。同一个工作流可反复应用到不同数据集上。

### 1.2 核心功能

- **工作流管理**：创建、编辑、删除、克隆工作流（工作流为独立模板，不绑定数据集）
- **节点编排**：添加、配置、排序、启用/禁用节点
- **应用到数据集**：将工作流应用到任意数据集，选择目标数据集和初始版本后执行
- **执行监控**：实时进度展示、节点级状态追踪、失败中断与错误报告
- **执行历史**：保存每次执行记录（含目标数据集信息），支持查看详情和结果追溯

### 1.3 技术栈

| 层级     | 技术                       | 说明                              |
| -------- | -------------------------- | --------------------------------- |
| 前端     | Vue 3 + Element Plus       | 工作流编辑器 + 执行视图           |
| 通信     | Electron IPC               | invoke / on 双向通信              |
| 后端     | TypeScript (Electron main) | Controller → Service → Repository |
| 执行引擎 | WorkflowService            | 线性节点调度，调用已有 Service    |
| 存储     | SQLite (better-sqlite3)    | 工作流定义 + 执行记录             |

### 1.4 设计原则

- **第一阶段：线性流水线**，节点按顺序依次执行
- **预留扩展接口**，后续可升级为可视化 DAG 节点图（类似 n8n / Node-RED）
- **复用现有 Service**，不重新实现业务逻辑，仅做参数适配和调度编排
- **数据版本自动传递**，上一个产出版本节点的输出自动成为下一个节点的输入

## 2. 需求分析

### 2.1 核心需求

| 需求编号 | 需求描述                                           | 优先级 |
| -------- | -------------------------------------------------- | ------ |
| WF-001   | 用户能创建工作流作为独立流程模板（不绑定数据集）   | P0     |
| WF-002   | 用户能在工作流中添加节点，选择节点类型             | P0     |
| WF-003   | 用户能为每个节点配置参数（复用现有功能的参数体系） | P0     |
| WF-004   | 用户能调整节点顺序（拖拽或上下移动）               | P0     |
| WF-005   | 用户能将工作流应用到指定数据集，节点按顺序自动运行 | P0     |
| WF-006   | 执行过程中显示实时进度（当前节点、百分比、状态）   | P0     |
| WF-007   | 执行失败时中断并显示错误信息，已完成节点的结果保留 | P0     |
| WF-008   | 用户能查看执行历史和每个节点的执行详情             | P1     |
| WF-009   | 用户能启用/禁用单个节点（跳过不需要的步骤）        | P1     |
| WF-010   | 用户能克隆已有工作流作为模板                       | P2     |

### 2.2 用户场景

#### 场景一：标准通量数据质控流水线

```
用户导入一年的通量数据后，创建工作流：
  节点1: 异常检测 (阈值过滤，应用物理范围限制)
  节点2: 缺失值插补 (REddyProc MDS 方法)
  节点3: 通量分割 (夜间法 Reichstein)
  节点4: 数据导出 (CSV 格式)

一键执行 → 自动完成整个 QC 流程
```

#### 场景二：多方法对比分析

```
用户想对比不同插补方法的效果：
  工作流A: 异常检测 → 线性插值 → 通量分割
  工作流B: 异常检测 → ARIMA插补 → 通量分割

分别执行两个工作流，对比最终结果
```

#### 场景三：日常数据更新（工作流复用）

```
新数据到来后，用户直接复用已有工作流：
  选择工作流 → 点击“应用到数据集” → 选择新数据集 + 版本 → 执行

无需重新配置任何参数，工作流作为模板直接复用
```

### 2.3 非功能需求

| 需求       | 说明                                   |
| ---------- | -------------------------------------- |
| 执行可靠性 | 单节点失败不影响已完成节点的结果       |
| 进度可观测 | 支持节点级和整体级进度展示             |
| 数据完整性 | 每个节点的输入/输出版本 ID 完整记录    |
| 性能       | 工作流定义加载 < 200ms，节点切换无卡顿 |

## 3. 功能设计

### 3.1 核心概念

```
┌─────────────────────────────────────────────────────────────┐
│              Workflow (工作流 = 流程模板)                    │
│  - 独立于数据集，不绑定特定数据                               │
│  - 包含 N 个有序节点 + 参数配置                               │
│  - 可反复应用到不同数据集上                                   │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐              │
│  │  Node 1  │ →  │  Node 2  │ →  │  Node 3  │ → ...        │
│  │ 异常检测  │    │ 缺失值插补 │    │ 通量分割  │              │
│  └──────────┘    └──────────┘    └──────────┘              │
│       │                │                │                    │
│   config_json      config_json     config_json              │
│                                                              │
├─────────────────────────────────────────────────────────────┤
│              Execution (执行记录)                              │
│  - 绑定目标数据集 + 初始版本（执行时指定）                      │
│  - 记录每次执行的整体状态                                      │
│  - 包含 N 个节点执行详情                                      │
│  - 记录每个节点的输入/输出版本                                  │
└─────────────────────────────────────────────────────────────┘
```

**术语定义**：

| 术语                           | 说明                                                |
| ------------------------------ | --------------------------------------------------- |
| **Workflow（工作流）**         | 用户创建的自动化处理流程模板，独立于数据集          |
| **Node（节点）**               | 工作流中的单个处理步骤，对应一个已有功能            |
| **Node Type（节点类型）**      | 节点的功能类别（异常检测 / 插补 / 通量分割等）      |
| **config_json**                | 节点的参数配置，JSON 格式存储                       |
| **Execution（执行记录）**      | 一次完整的工作流执行过程，绑定目标数据集 + 初始版本 |
| **Node Execution（节点执行）** | 一个节点在某次执行中的运行详情                      |

### 3.2 节点类型定义

| 节点类型标识           | 显示名称   | 图标 | 产出新数据版本 | 对应后端 Service             |
| ---------------------- | ---------- | ---- | -------------- | ---------------------------- |
| `OUTLIER_DETECTION`    | 异常检测   | 🔍   | ✅             | `OutlierDetectionService`    |
| `IMPUTATION`           | 缺失值插补 | 🔧   | ✅             | `ImputationService`          |
| `FLUX_PARTITIONING`    | 通量分割   | 📊   | ✅             | `FluxPartitioningService`    |
| `CORRELATION_ANALYSIS` | 相关性分析 | 🔗   | ❌（仅分析）   | `CorrelationAnalysisService` |
| `EXPORT`               | 数据导出   | 📤   | ❌（仅导出）   | `ExportService`              |

**节点分类**：

- **数据转换节点** (Transform)：执行后产生新的数据版本（异常检测、插补、通量分割）
- **分析节点** (Analysis)：执行分析但不改变数据（相关性分析）
- **输出节点** (Output)：将数据导出到外部（数据导出）

### 3.3 数据流模型

数据版本在节点间的传递是自动化工作流的核心机制。执行时由用户指定目标数据集和初始版本：

```
执行开始（用户选择目标数据集 + 初始版本）
    │
    ▼
initial_version_id (执行时指定的起始版本)
    │
    ▼
┌──────────────────────────────────────────────┐
│  Node 1: 异常检测                             │
│  input_version_id  = initial_version_id       │
│  执行异常检测 + 应用过滤                        │
│  output_version_id = 新生成的 version_id (v2)  │
└──────────────────────────────────────────────┘
    │
    ▼  currentVersionId = v2
    │
┌──────────────────────────────────────────────┐
│  Node 2: 缺失值插补                           │
│  input_version_id  = v2                       │
│  执行插补                                     │
│  output_version_id = 新生成的 version_id (v3)  │
└──────────────────────────────────────────────┘
    │
    ▼  currentVersionId = v3
    │
┌──────────────────────────────────────────────┐
│  Node 3: 通量分割                             │
│  input_version_id  = v3                       │
│  执行通量分割                                  │
│  output_version_id = 新生成的 version_id (v4)  │
└──────────────────────────────────────────────┘
    │
    ▼  currentVersionId = v4
    │
┌──────────────────────────────────────────────┐
│  Node 4: 数据导出                             │
│  input_version_id  = v4                       │
│  导出到文件                                    │
│  output_version_id = null (不产生新版本)        │
└──────────────────────────────────────────────┘
    │
    ▼
工作流完成
```

**规则**：

1. 数据转换节点执行后产生新版本，`currentVersionId` 更新为新版本 ID
2. 分析/输出节点不改变 `currentVersionId`，下一个节点继续使用当前版本
3. 禁用的节点（`is_enabled = false`）被跳过，不影响版本传递
4. 节点失败时立即中断，后续节点不再执行

### 3.4 工作流状态机

```
                    ┌──────────┐
         创建       │  DRAFT   │  编辑中，可随时修改
                    └────┬─────┘
                         │ 用户配置完成
                         ▼
                    ┌──────────┐
                    │  READY   │  就绪，可以执行
                    └────┬─────┘
                         │ 用户点击执行
                         ▼
                    ┌──────────┐
          ┌────────│ RUNNING  │  执行中
          │        └────┬─────┘
          │             │
     执行失败      执行成功
          │             │
          ▼             ▼
     ┌──────────┐ ┌──────────┐
     │  FAILED  │ │COMPLETED │
     └────┬─────┘ └────┬─────┘
          │             │
          └──────┬──────┘
                 │ 用户再次编辑或重新执行
                 ▼
            ┌──────────┐
            │  READY   │
            └──────────┘
```

### 3.5 节点执行状态机

```
PENDING → RUNNING → COMPLETED
                  → FAILED
                  → SKIPPED (节点被禁用时)
```

### 4.4 与现有服务的集成

```typescript
// 伪代码：节点执行器映射
const nodeExecutors: Record<WorkflowNodeType, NodeExecutor> = {
  OUTLIER_DETECTION: {
    async execute(config, inputVersionId, datasetId) {
      // 1. 调用 OutlierDetectionService.executeThresholdDetection()
      // 2. 调用 OutlierDetectionService.applyFiltering()
      // 3. 返回新版本 ID
    },
  },
  IMPUTATION: {
    async execute(config, inputVersionId, datasetId) {
      // 1. 调用 ImputationService.executeImputation()
      // 2. 调用 ImputationService.applyVersion()
      // 3. 返回新版本 ID
    },
  },
  FLUX_PARTITIONING: {
    async execute(config, inputVersionId, datasetId) {
      // 1. 调用 FluxPartitioningService.execute()
      // 2. 返回新版本 ID
    },
  },
  CORRELATION_ANALYSIS: {
    async execute(config, inputVersionId, datasetId) {
      // 1. 调用 CorrelationAnalysisService.analyze()
      // 2. 返回 null (不产生新版本)
    },
  },
  EXPORT: {
    async execute(config, inputVersionId, datasetId) {
      // 1. 调用 ExportService.execute()
      // 2. 返回 null (不产生新版本)
    },
  },
};
```

## 5. 节点配置规范

每种节点类型的 `config_json` 有固定的 JSON Schema，下面定义各类型的配置结构。

### 5.1 异常检测节点

```typescript
interface OutlierDetectionNodeConfig {
  // 检测方法
  detectionMethod: "THRESHOLD_STATIC" | "ZSCORE" | "IQR";
  // 目标列（null 表示全部数值列）
  targetColumns: string[] | null;
  // 方法参数
  methodParams: {
    // THRESHOLD_STATIC 时使用
    usePhysicalRange?: boolean;
    // ZSCORE 时使用
    zscoreThreshold?: number; // 默认 3.0
    // IQR 时使用
    iqrMultiplier?: number; // 默认 1.5
  };
  // 是否自动应用过滤（生成新版本）
  autoApply: boolean; // 默认 true
}
```

### 5.2 缺失值插补节点

```typescript
interface ImputationNodeConfig {
  // 插补方法
  methodId: string; // 如 'LINEAR', 'ARIMA', 'MDS_REDDYPROC'
  // 目标列（null 表示全部含缺失值的列）
  targetColumns: string[] | null;
  // 方法参数（与现有 ImputationService 参数体系一致）
  methodParams: Record<string, any>;
  // 是否自动应用到数据集（生成新版本）
  autoApply: boolean; // 默认 true
}
```

### 5.3 通量分割节点

```typescript
interface FluxPartitioningNodeConfig {
  // 分割方法
  methodId: "NIGHTTIME_REICHSTEIN" | "DAYTIME_LASSLOP";
  // 列映射
  columnMapping: {
    NEE: string;
    Rg: string;
    Tair: string;
    VPD: string;
    Ustar?: string;
  };
  // 站点信息
  siteInfo: {
    latitude: number;
    longitude: number;
    timezone: number;
  };
  // 可选项
```

## 7. 类型定义

完整的 TypeScript 类型定义将放在 `shared/types/workflow.ts`，核心类型概要如下：

```typescript
// ============ 节点类型枚举 ============
export type WorkflowNodeType =
  | "OUTLIER_DETECTION"
  | "IMPUTATION"
  | "FLUX_PARTITIONING"
  | "CORRELATION_ANALYSIS"

export type WorkflowStatus = "DRAFT" | "READY" | "RUNNING" | "COMPLETED" | "FAILED";
export type NodeExecutionStatus = "PENDING" | "RUNNING" | "COMPLETED" | "FAILED" | "SKIPPED";
export type ExecutionStatus = "PENDING" | "RUNNING" | "COMPLETED" | "FAILED" | "CANCELLED";

// ============ 工作流定义（流程模板，不绑定数据集） ============
export interface Workflow {
  id: number;
  name: string;
  description?: string;
  datasetId: number | null; // v2.0: 改为可空，不再必填
  initialVersionId: number | null; // v2.0: 改为可空，不再必填
  status: WorkflowStatus;
  createdAt: string;
  updatedAt: string;
}

// ============ 工作流节点 ============
export interface WorkflowNode {
  id: number;
  workflowId: number;
  nodeOrder: number;
  nodeType: WorkflowNodeType;
  nodeName: string;
  configJson: string; // JSON 字符串
```

## 8. 扩展预留

### 8.1 节点执行器接口

定义标准化接口，便于后续新增节点类型：

```typescript
interface INodeExecutor {
  // 节点类型标识
  readonly nodeType: WorkflowNodeType;
  // 验证配置是否合法
  validateConfig(config: any): { valid: boolean; errors?: string[] };
  // 执行节点
  execute(context: NodeExecutionContext): Promise<NodeExecutionResult>;
}

interface NodeExecutionContext {
  datasetId: number;
  inputVersionId: number;
  config: any;
  onProgress: (progress: number, message: string) => void;
}

interface NodeExecutionResult {
  outputVersionId: number | null;
  resultData?: any;
}
```

### 8.2 可视化节点图预留

数据库表 `biz_workflow_node` 预留 `position_x` 和 `position_y` 字段（默认 NULL），当前线性模式下不使用，后续升级为可视化节点图时用于存储节点在画布上的位置。

### 8.3 执行策略扩展

当前执行引擎为线性顺序执行，后续可扩展为：

| 策略          | 说明                                     |
| ------------- | ---------------------------------------- |
| `LINEAR`      | 当前默认，按 node_order 顺序执行         |
| `PARALLEL`    | 支持并行分支（需 DAG 支持）              |
| `CONDITIONAL` | 支持条件分支（根据上一节点结果决定路径） |

### 8.4 工作流模板（v2.0 已实现）

v2.0 中工作流已重构为独立流程模板，创建时不绑定数据集，执行时由用户选择目标数据集和初始版本。同一工作流可反复应用到不同数据集上。
